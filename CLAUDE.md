# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is a fork of the [Helix editor](https://github.com/helix-editor/helix) (`origin` is `kilianbartz/helix-jupyter`) that adds a **Jupyter REPL** feature: evaluate buffer selections in a persistent Jupyter kernel and render output inline. The Jupyter work is the reason this fork exists — when working here, changes will almost always touch the Jupyter integration described below. The rest is upstream Helix.

`REPL.md` is the user-facing documentation for the Jupyter feature and is the authoritative spec for its behavior, commands, config keys, and limitations. Read it before changing REPL behavior.

`LATEX.md` is the user-facing documentation for the LaTeX section-navigation feature (section jumps, `@section.around` captures, `goto_next/prev_section` commands). Read it before changing LaTeX textobject queries or the section-jump commands.

## Build, test, run

The toolchain is pinned in `rust-toolchain.toml` (Rust 1.90.0, the project's MSRV). `default-members` is `helix-term`, so plain cargo commands target the editor binary `hx`.

```sh
cargo run -- <file>            # run the debug editor (fast iteration)
cargo build --release          # release build (binary at target/release/hx)
cargo test --workspace         # unit + doc tests across all crates
cargo integration-test         # helix-term integration tests (alias in .cargo/config.toml)
cargo xtask docgen             # regenerate autogenerated docs (command/lang lists in book/)
```

Notes:
- `cargo integration-test` is an alias for `test --features integration --profile integration --workspace --test integration`. Integration tests require the `integration` feature.
- Run a single test by name: `cargo test --workspace <substring>`, or for integration tests `cargo integration-test <substring>`.
- `HELIX_LOG_LEVEL=debug` controls log verbosity in integration tests. For the running editor, `cargo run -- --log foo.log <file>` writes a log you can `tail -f`; print-debug with `log::info!`/`warn!`/`error!`.
- `runtime/` (grammars, queries, themes) must be discoverable at runtime. `cargo run` from the repo root finds it via the workspace; the `.envrc`/nix shell sets `HELIX_RUNTIME`.

### Jupyter-specific testing

```sh
# Smoke tool: spawn a kernel and print its output
cargo run -p helix-jupyter --example spike -- <kernelspec-name>

# Jupyter integration tests — require a kernelspec named `helix-test`;
# they skip themselves if it is absent
cargo test -p helix-term --features integration --test integration jupyter
```

To exercise the feature interactively you need a discoverable Jupyter kernelspec (e.g. install `ipykernel` and register it). See `REPL.md` → Prerequisites.

## Workspace architecture

Helix is split into layered crates (see `docs/architecture.md` for the full overview):

- **helix-core** — functional editing primitives: `Rope` (re-exported ropey), `Selection`/`Range`, and OT-like `Transaction`s. Operations return new copies rather than mutating. `Transaction::change`/`change_by_selection` is the main edit interface.
- **helix-view** — the imperative shell. `Editor` holds global state (all open documents, the view tree, config, LSP/DAP/Jupyter registries). `Document` ties a `Rope` + `Selection`s + `Syntax` + `History` together. `View` is an open split.
- **helix-term** — the terminal frontend. `main` sets up `Application`, which runs the event loop. `commands/` holds all bindable actions; `keymap.rs` binds keys to them; `ui/` holds `Component`s rendered by the `Compositor`.
- **helix-lsp** / **helix-dap** — LSP and Debug Adapter Protocol clients. The Jupyter integration is deliberately modeled on the **DAP** integration, not LSP (LSP has no concept of code execution).
- **helix-event** — event primitives (document open/change/close, server start/stop, etc.).
- **helix-tui**, **helix-loader**, **helix-vcs**, **helix-stdx**, **helix-parsec** — TUI primitives, resource loading, version control, stdlib extensions, parser combinators.

## Jupyter feature architecture

The feature spans three crates and is structured to mirror `helix-dap`:

**`helix-jupyter/`** — the kernel client crate. Knows nothing about the editor.
- `client.rs` — `Client` wraps a single running kernel.
- `registry.rs` — `Registry` owns all kernels (keyed by `KernelId`) and exposes a merged stream of incoming messages.
- `messages.rs` — `Payload` (editor-facing message enum), `Channel`, `media_to_text`.
- `lib.rs` — re-exports protocol types from `jupyter-protocol`. The ZeroMQ wire protocol, multipart framing, and HMAC signing are handled by the `jupyter-zmq-client` and `jupyter-protocol` external crates — do not reimplement them.

**`helix-view/`** — editor state and the async message handler.
- `jupyter.rs` — per-document state: `JupyterOutput`, `OutputKind`, `ExecutionState`, `strip_ansi`, and the `Editor`/`Document` fields. A kernel is associated **per document**.
- `handlers/jupyter.rs` — `Editor::handle_jupyter_message`, which mirrors `Editor::handle_debugger_message`. Kernel messages flow back through the normal editor event loop (dispatched in `helix-term/src/application.rs`) and mutate output state, returning `true` when a re-render is needed.

**`helix-term/`** — commands and rendering.
- `commands/jupyter.rs` — the `:jupyter-*` typed commands and `jupyter_*` static (bindable) commands. New commands must also be wired in `commands/typed.rs`.
- `ui/text_decorations/jupyter.rs` — renders evaluation output as virtual lines beneath the evaluated code. Output is anchored to a line and re-positioned across edits, the same mechanism diagnostics use.

Key behavioral facts (see `REPL.md` for the rest):
- Eval operates on **whole lines** touched by the selection; a bare cursor evaluates its current line.
- Variable inspection is **Python-specific** (it injects a small Python probe expression after each eval).
- Config lives under `[editor.jupyter]` (`enable`, `default-kernel`, `auto-start`, `inline-output`, `max-output-lines`, `inspect-variables`).
- Theme scopes: `ui.virtual.jupyter.output` and `ui.virtual.jupyter.error`.

## Code folding feature architecture

`FOLDING.md` is the user-facing documentation for the folding feature and is the authoritative spec for its commands, keybindings, theming, and limitations.

The feature is implemented across three layers:

**`helix-core/src/text_annotations.rs`** — `FoldSpan` struct: a `start`/`end` char-index pair plus a cached `end_line`. Folds are fed into the existing `TextAnnotations` pipeline (the same one used by inlay hints and diagnostics), so rendering requires no special-casing in the formatter.

**`helix-view/src/document.rs`** — `Document` holds `folds: Vec<FoldSpan>`. On every `apply` the spans are position-tracked (`Assoc::After`/`Before`) and folds whose body shrinks to zero lines are dropped. `normalize_folds` keeps the list sorted by `start` and resolves overlaps (the earlier/outer fold wins).

**`helix-term/src/commands.rs`** — the five fold commands (`fold_close`, `fold_open`, `fold_toggle`, `fold_all`, `unfold_all`) and their typed counterparts in `commands/typed.rs`. `fold_all` uses tree-sitter `textobjects.scm` captures (`@function`, `@class`) to find foldable ranges. Default keybindings (`za`, `zf`, `zo`, `zM`, `zR`) are in `keymap/default.rs` under the `z`/`Z` view menu.

Key behavioral facts (see `FOLDING.md` for the rest):
- Folding is tree-sitter driven — it works for any language whose `textobjects.scm` defines `function`/`class` captures (Python, the REPL target language, is included).
- Folds are stored per-document as a sorted, non-overlapping `Vec<FoldSpan>` and are not persisted across editor sessions.
- The signature line (first line of the block) stays visible; all subsequent lines of the block are concealed and replaced with a `…` marker.
- Theme scope: `ui.virtual.fold` (controls the `…` marker color and underline).

## Spell-checking LSP architecture

`SPELL.md` is the user-facing documentation for the spell checker and is the authoritative spec for its config keys, dictionaries, code actions, and limitations.

Unlike folding and Jupyter, this feature is **not** built into the editor. It is a standalone language server in the **`helix-spell-lsp/`** subdirectory, wired into the editor only through `languages.toml` (like `texlab`/`marksman`). The crate is deliberately **detached from the Helix workspace** (its `Cargo.toml` has an empty `[workspace]` table, and it is not in the root `members`) so its `zspell`/tree-sitter dependencies and C-grammar build never enter the editor's build.

- `src/extract.rs` — tree-sitter extraction. Uses `tree-sitter-md` (Markdown), the LaTeX grammar (latex-lsp), and the Typst grammar (uben0/tree-sitter-typst). Strategy for LaTeX/Markdown: collect byte ranges of non-prose nodes (commands, math, code, links) into a merged skip-list, then tokenize the whole document and drop words overlapping a skipped range. Typst inverts this: prose lives only in `text` (and `quote`) nodes, and code blocks can embed checkable markup `content`, so it collects those nodes as an *include* list and skips the complement.
- `src/dictionary.rs` — `zspell` base dictionary plus two in-memory word sets (project `.spell.dic`, personal `~/.config/helix-spell/personal.dic`); adding a word is an O(1) insert + file append, never a dictionary rebuild.
- `src/backend.rs` — the `tower-lsp` `LanguageServer`: FULL-sync diagnostics, quick-fix code actions, and the `helix-spell.addWord` command.
- `build.rs` — compiles the LaTeX and Typst grammars' `parser.c` + `scanner.c` (see the `GRAMMARS` table). The published `tree-sitter-latex` 0.1.0 crate is unusable (ships `parser.c` without its external `scanner.c`), so the grammars are instead compiled from the repo's vendored `runtime/grammars/sources/{latex,typst}/src` (override with `HELIX_SPELL_LATEX_SRC` / `HELIX_SPELL_TYPST_SRC`). Those paths are `.gitignored` and populated by the grammar fetch step, so the editor's grammars must be fetched before building this crate.

Key behavioral facts (see `SPELL.md` for the rest):
- Registered for LaTeX, Markdown and Typst with `only-features = ["diagnostics", "code-action"]` so it never competes with `texlab`/`marksman`/`tinymist`.
- Config lives under the server's `config` table in `languages.toml` (forwarded as `initializationOptions`): `dictionary`, `aff-path`/`dic-path`, `project-dict-file`, `severity`, `ignore-uppercase`, `max-suggestions`.
- Suggestions are best-effort (zspell's edit-distance engine); this adds no `:command`, so no `cargo xtask docgen` is needed.

## Conventions

- After adding or changing a `:command`, run `cargo xtask docgen` to regenerate the docs in `book/` (the command and language tables are autogenerated).
- `rustfmt.toml` is intentionally empty — use default `cargo fmt`. Clippy and rustfmt are part of the pinned toolchain.
- MSRV bumps must be updated in three places: `workspace.package.rust-version` in `Cargo.toml`, `env.MSRV` in `.github/workflows/build.yml`, and `toolchain.channel` in `rust-toolchain.toml`.
